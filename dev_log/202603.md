# **0306-ASR Pipeline Debug 全程复盘**

今天从零开始搭建了一套完整的中文多说话人 ASR 系统，中间踩了不少坑，下面按时间线完整复盘。

## **一、系统架构确定阶段**

最终选定的技术栈是 **faster-whisper + pyannote.audio + MockAudioDecoder**，三者分工明确：faster-whisper 负责转写和时间戳，pyannote 负责说话人分离，MockAudioDecoder 是为了绕过 torchcodec 的兼容性问题而引入的 soundfile 替代层。

这个架构的核心设计原则是**松耦合**——ASR 和 Diarization 两个模块独立运行，最后通过 `_merge_diarization` 合并结果，任何一个模块都可以单独替换（比如后续换成 FunASR）而不影响另一个。

## **二、环境依赖问题：torchcodec 加载失败**

**现象：** 启动时报 `OSError: Could not load this library: libtorchcodec_core4.so`

**根因分析：**
- 环境是 CUDA 13.0 + PyTorch 2.9.1，而 torchcodec 0.10.0 安装的是无 CUDA 后缀的纯净版
- torchcodec 官方目前最高只发布到 `cu126`，没有 `cu130` 的预编译包，版本完全不匹配
- `pip show torchcodec` 的 `Required-by` 字段显示是 **pyannote-audio** 把它拉进来的

**解决方案：** 直接 `pip uninstall torchcodec`，pyannote 的实际推理路径完全不依赖视频解码，卸载后功能不受影响。

**经验：** 遇到 `.so` 加载失败，第一步永远是用 `pip show <package> | grep Required-by` 找到是谁引入了这个依赖，再判断是否真的需要它。CUDA 版本越新，越容易遇到预编译包缺失的问题。

## **三、音频预处理问题：格式转换引发质量下降**

**现象：** 将 MP4 转换为 16kHz WAV 后，ASR 识别内容大量丢失

**排查过程：**

第一步排查文件完整性，`ffprobe` 确认转换后文件 `duration=599.9s`，和原始 MP4 完全一致，文件没有损坏。

第二步对比 whisperx 内部转换和手动转换的结果，发现两者输出一致，排除了转换质量差异。

第三步关掉 `vad_filter` 后内容恢复，**确认问题根本不在格式转换，而在 VAD 参数**。

**真正的根因：** faster-whisper 默认 `vad_filter=False`，没有 VAD 时模型用固定 30 秒窗口切分音频，静音段触发大量幻觉输出，同时 `condition_on_previous_text=True` 导致幻觉错误像滚雪球一样传染后续片段。

**解决方案：**
```python
segments_generator, info = model.transcribe(
    audio_path,
    language="zh",
    beam_size=5,
    vad_filter=True,
    vad_parameters=dict(
        threshold=0.3,              # 默认 0.5 太严格，调低避免误判语音
        min_speech_duration_ms=100,
        min_silence_duration_ms=300,
        speech_pad_ms=400,
    ),
    condition_on_previous_text=False,  # 长音频必须关闭
    no_speech_threshold=0.8,
)
```

**经验：** 音频预处理和模型参数是两个独立的问题，排查时要用**控制变量法**逐步隔离，不要同时改多个变量。faster-whisper 直接接受原始 MP4 路径，内部会自动处理格式，不需要手动预处理。

## **四、MockAudioDecoder 数据正确性验证**

由于 torchcodec 被卸载，pyannote 的 AudioDecoder 需要用 soundfile 替代，验证 mock 是否正确的方法：

```python
decoder = MockAudioDecoder("audio.wav")
samples = decoder.get_samples_played_in_range(0, 5)
# 检查三个关键指标
assert samples.data.shape == (1, 80000)           # 单声道 × 5s × 16kHz
assert -1.0 <= samples.data.min() <= samples.data.max() <= 1.0  # 值域正确
assert decoder.metadata.sample_rate == 16000       # 采样率正确
```

**关键细节：** `sf.read` 必须指定 `dtype="float32"`，否则某些格式会返回 int16 值域（-32768\~32767），传给 pyannote 会导致说话人分离结果完全错误。

## **五、音频基础知识梳理**

本次 debug 过程中涉及的音频基础概念：

**采样率（Sample Rate）**是每秒采集的音频样本数，单位 Hz。Whisper 和 FunASR 都要求 16000Hz。原始 MP4 通常是 44100Hz（CD 音质），降采样到 16000Hz 时需要使用高质量重采样算法（SoXR），但对语音 ASR 来说 ffmpeg 默认的 SWR 引擎已经足够。

**声道（Channel）**是一条音轨内的独立音频信号。单声道（mono）= 1 个信号，立体声（stereo）= 左右 2 个信号。ASR 模型要求单声道输入，双声道需要 downmix。判断是否需要拆开处理的方法是计算左右声道的皮尔逊相关系数，相关系数 > 0.9 说明内容相同可以直接 downmix，< 0.7 说明是双轨录制需要拆开。

**音轨（Audio Track）**是文件容器层面的概念，一个 MP4 可以有多条音轨（如普通话版、英文配音版）。声道是单条音轨内部的信号数量，两者是不同层级的概念。

**VAD（Voice Activity Detection）**是语音活动检测，用于区分有人声和无声的片段。Silero VAD 是 faster-whisper 内置的 VAD 模型，`threshold` 参数控制判断灵敏度，值越低越宽松（保留更多片段），值越高越严格（丢弃更多片段）。对于播客类音频，0.3\~0.4 是合适的范围。

**幻觉（Hallucination）**是生成式模型在处理静音或低信噪比片段时凭空生成不存在内容的现象，是 Whisper 最常见的质量问题。开启 VAD 是最有效的抑制手段，`no_speech_threshold` 和 `compression_ratio_threshold` 是辅助手段。

## **六、模型选型知识沉淀**

| 对比维度 | faster-whisper large-v3 | FunASR SenseVoice |
|---|---|---|
| 架构 | 生成式 Encoder-Decoder | 非自回归 CTC |
| 中文 CER | \~4.8% | \~2.6% |
| 口语/语气词 | 倾向过滤，输出更书面 | 忠实还原，保留语气词 |
| 笑声处理 | 跳过或输出 `[笑声]` | 输出 `<laughter>` 标签 |
| 标点恢复 | 无（需后处理） | 内置 CT-Transformer |
| 长音频稳定性 | 需关闭 condition_on_previous_text | 内置 VAD，天然稳定 |
| 说话人分离 | 需外接 pyannote | 需外接 pyannote |
| 幻觉风险 | 较高（静音段） | 较低（非自回归） |

**评估两个模型哪个更好的方法**是准备 3\~5 段有代表性的音频片段，人工转写作为 Ground Truth，用 `jiwer` 库计算 CER，同时人工评估语气词保留度、笑声标注、断句自然度等 CER 无法覆盖的维度。

## **七、核心 Debug 方法论总结**

今天整个排查过程体现了几个通用的 debug 原则：

**控制变量**：每次只改一个参数，对比前后结果。不要同时修改格式转换 + 模型参数 + VAD 设置，否则无法定位是哪个变量起了作用。

**加调试日志**：在 pipeline 的每个关键节点打印数据量（片段数、字符数、时间戳范围），内容在哪一步开始减少，问题就在哪一步。

**验证数据而不是假设**：不要假设转换后的音频是正确的，用 `ffprobe` 和统计信息实际验证。不要假设 mock 的数据格式正确，用 shape 和值域实际验证。

**从最简单的假设开始**：遇到 `.so` 加载失败，先检查是否真的需要这个库；遇到内容丢失，先关掉所有过滤参数看是否恢复，再逐步加回来。
